The Protocol: Introducing the COSTAR Framework

Local AI models are powerful, but unguided, they are chaotic engines of pure probability. To transform raw model output into usable educational architecture, we need a structured protocol of communication.

  • C - CONTEXT (The World State) Define the environment, the constraints, and the current reality the AI is operating within. Without context, the AI is hallucinating in a void.
  • O - OBJECTIVE (The Quest) Be ruthlessly precise. Vague objectives yield vague results. "Write code" is bad; "Create a Rust struct for player health" is good.
  • S - STYLE (The Format) Do you need a Python script, a JSON object, a Socratic dialogue, or a Markdown table? Define the architectural blueprint.
  • T - TONE (The Vibe) Crucial for NPCs. Is the AI a grumpy dwarven blacksmith, an encouraging tutor, or a coldly logical computer terminal?
  • A - AUDIENCE (The Learner) Who is consuming this output? Tailor the complexity accordingly.
  • R - RESPONSE (The Constraints) Specify output length, forbidden words, required libraries, or formatting rules.

Use Case 1: The Socratic AI NPC

Moving beyond "Click A, B, or C." In a Sovereign Sandbox, assessment is a dialogue. By embedding a local LLM into your Bevy game via WASM, you can create NPCs that don't just dispense lore—they gauge understanding through conversation.

[CONTEXT] You are Chronos, an ancient, slightly damaged AI historian construct embedded in the walls of a virtual Roman Forum simulation. The player is a student explorer. They have just examined a digital bust of Cicero.

[OBJECTIVE] Engage the player in conversation about the Roman Senate. Use the Socratic method to guide them toward realizing the difference between the Roman Republic and a modern democracy.

[STYLE] A dialogue screenplay format between [CHRONOS] and [PLAYER]. Keep responses under 3 sentences.

[TONE] Ancient, wise, slightly cryptic, and encouraging. Use metaphors related to time, data corruption, and stone.

[AUDIENCE] A 10th-grade history student who is easily bored by lectures but enjoys puzzles.

[RESPONSE] Provide only the initial greeting. Do not break character. Return as JSON {"dialogue": string}

Use Case 2: Generating Game Logic (Rust/Bevy)

The biggest barrier for educators wanting to build games is the terror of the blank code editor. As a "Local AI Architect," you use AI integrated into your IDE to act as a "Senior Engineer" sitting on your shoulder.

[CONTEXT] I am working in a Rust Bevy project (version 0.13). I have an existing 3D game setup with a player character controller.

[OBJECTIVE] Write a Bevy ECS System using Rust. This system should spawn five rotating "Knowledge Fragment" glowing cubes at random locations (x: -10 to 10, z: -10 to 10). They need a component to identify them as collectible.

[STYLE] Clean, commented Rust code. Use standard Bevy patterns. Provide the component struct definition and the spawning system function.

[TONE] Professional, technical, and highly efficient senior engineer.

[AUDIENCE] A beginner Bevy developer who understands basic ECS concepts but struggles with syntax.

[RESPONSE] Provide only the necessary code blocks. Ensure the code is copy-paste ready inside ` ```rust ` blocks.

The Sovereign Sandbox Principles

I. Architectures of Curiosity over Compliance

Traditional education forces compliance. Sandboxes foster intrinsic motivation. Design systems where the AI rewards curiosity. The "test" shouldn't look like a test; it should look like a locked door the player genuinely wants to open.

II. Dynamic Scaffolding (The Infinite Vygotsky)

Local AI is the ultimate, scalable "More Knowledgeable Other." It never gets tired, never judges, and can infinitely adjust its scaffolding based on the COSTAR prompts you design to meet the learner exactly where they are.

III. Data Sovereignty is Pedagogical Power

By running models on the user's own hardware, we ensure total privacy. There are no subscriptions, no data scraping, and no latency. This sovereignty allows educators to build safe, private spaces for experimentation.
